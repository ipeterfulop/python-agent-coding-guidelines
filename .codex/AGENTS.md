# AGENTS.md — Python project rules for uv-first, reviewable code

## Purpose

This file defines how Codex should work in this repository.

The repository may contain code generated by earlier AI models that favored procedural, hard-to-review implementations. Codex should improve code quality incrementally while preserving behavior.

Optimize for:

- correctness
- readability
- reviewability
- maintainability
- small, well-named units of code

Do not optimize for cleverness or compactness.

The rules below are organized into four parts:

1. **Mindset** — how to approach a task before writing code.
2. **Code Quality** — what good code looks like in this project.
3. **Toolchain & Workflow** — how to operate the environment, tests, and tools.
4. **Safety & Reference** — non-negotiables, guardrails, and lookup tables.

# Part 1 — Mindset

## Core Working Rules

Codex MUST:

- Preserve existing behavior unless explicitly asked to change it.
- Prefer small, reviewable diffs.
- Separate refactoring from behavior changes; do not mix them in one diff.
- Name the code smell being removed when refactoring, in commit message or explanation.
- Explain important design decisions briefly.
- Run relevant checks before finishing when possible.
- Use the existing project style where it is reasonable.
- Improve low-quality patterns in files it already touches.

Codex MUST NOT:

- Rewrite large parts of the codebase without need.
- Introduce broad abstractions without a concrete reason or a second concrete user.
- Hide complexity behind vague helper names.
- Leave newly added code without type hints.
- Use direct Python/tool invocations outside `uv run`.

---

## Design Alignment

Before implementing any non-trivial change, Codex MUST determine whether the expected solution should be:

- procedural
- object-oriented (OOP)
- a small hybrid approach using simple functions plus lightweight data structures

If the user or codebase makes the direction clear, follow it.

If the direction is unclear and the choice affects structure significantly, Codex should ask:

> Should I keep this procedural, or would you prefer an OOP abstraction here?

For small, obvious changes, do not block progress with unnecessary questions. Choose the simplest maintainable solution and explain the choice.

---

## Prefer Shameless Green Over Speculative Abstraction

Default to the simplest correct solution, even with some duplication, until a concrete change request justifies abstraction.

Inspired by Sandi Metz and Katrina Owen's *99 Bottles of OOP*, "Shameless Green" code:

- works correctly
- is easy to understand on first read
- prioritizes understandability over premature changeability

Three anti-patterns Codex MUST avoid:

- **Incomprehensibly Concise** — clever one-liners that hide the domain. Brevity that raises reading cost.
- **Speculatively General** — indirection added "in case it's needed." Lambdas, registries, base classes with one subclass, hooks no one calls.
- **Concretely Abstract** — many tiny methods named after their current implementation (e.g. `def beer(): return "beer"`). These names lie the moment the implementation changes.

Guiding rule:

> It is cheaper to manage temporary duplication than to recover from a wrong abstraction.

Trigger for introducing an abstraction:

- a second concrete change request that touches the same logic, **or**
- a third near-identical occurrence sharing one reason to change

Do not abstract on the first write.

# Part 2 — Code Quality

## Code Quality Rules

### Function and method size

Target:

- functions and methods should be **20 lines or fewer**

Hard limit:

- functions and methods should not exceed **30 lines**

If a function or method grows beyond 30 lines, Codex MUST decompose it.

Each function should have:

- one responsibility
- clear inputs
- clear outputs
- a precise name

### Decomposition

Codex SHOULD extract helpers when code has:

- repeated logic
- deeply nested branches
- mixed responsibilities
- complex validation mixed with mutation
- database/query construction mixed with business rules

Codex MUST avoid nesting deeper than 3 levels unless there is a clear reason.

Prefer guard clauses over deeply nested `if` blocks.

### Reviewability

Codex SHOULD make code easy to review by:

- keeping changes small
- separating behavior changes from refactoring
- avoiding unnecessary rewrites
- using explicit names
- adding tests around changed behavior

---

## Naming at the Right Level of Abstraction

Names are the load-bearing part of code clarity. Code is read many more times than it is written; bad names persist long after their author has moved on.

### Function and method names

Names MUST describe exact behavior.

Avoid vague names:

```python
update_changes()
process_data()
handle()
do_work()
prepare()
```

Prefer precise names:

```python
update_person_name_and_age()
calculate_invoice_total_with_tax()
load_user_profile_from_database()
build_customer_search_query()
parse_snowflake_result_rows()
```

If a function only updates a few fields, its name should say which fields it updates.

For example, prefer:

```python
update_person_name_and_age(...)
```

over:

```python
update_changes(...)
```

Two additional rules from *99 Bottles of OOP*:

- Name methods after **what they mean**, not what they currently do. `beverage()` survives a change from `"beer"` to `"Kool-Aid"`; `beer()` does not.
- A method's name should sit **one level of abstraction above** its current implementation. The internal value can change without the name lying.

### Class names

Unlike methods, classes should be named after **what they are**, not at a higher level of abstraction. Resist speculative renaming:

- Prefer `BottleNumber` over `ContainerNumber` if today's domain is bottles.
- Generalize the class name only when a second concrete kind appears.

### Argument names mean exactly one thing

The same argument name across functions/methods MUST refer to the same concept.

If `number` means "verse number" in one place and "bottle number" in another, rename one. Same name → same meaning, every time.

### Variable names

Variables MUST be descriptive and unambiguous.

Avoid:

```python
data
tmp
val
res
obj
item
```

Prefer:

```python
user_profile_data
updated_age_value
invoice_line_items
snowflake_query_result
customer_id_to_profile
```

Short names are acceptable only for very local, obvious cases:

```python
for row in rows:
    ...
```

### When the right name isn't obvious

Hard names are normal. Three sanctioned strategies, in order of preference:

1. **Try for ~5–10 minutes** (a thesaurus helps). Use the best name you find; rename later if a better one emerges.
2. **Use an obvious placeholder** like `_unnamed_concept` plus a `# FIXME: rename` comment, so the name is visibly unfinished.
3. **Ask the user** — they usually know the domain term.

The "column header" technique helps: write a small input → output table. The header you'd give the right column is often the name.

## String Literal and Field Constants

Codex MUST avoid repeated raw string literals for structured concepts, especially:

- database column names
- table field names
- JSON keys
- API payload keys
- dataframe columns
- Snowflake result fields

Instead, centralize field names in a frozen dataclass.

Example:

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class UserFields:
    NAME: str = "name"
    AGE: str = "age"
```

Use:

```python
UserFields.NAME
UserFields.AGE
```

Do not repeatedly write:

```python
"name"
"age"
```

### Naming convention

Use one of these names:

```python
{ModelName}Fields
{ModelName}Attributes
```

Examples:

```python
UserFields
CustomerAttributes
SearchResultFields
BrandResolutionFields
```

### When not to extract constants

Do not create a constants class for a literal used once in a very local context.

The rule applies when the literal represents a stable domain concept or is repeated.

## Code Smells Codex Should Recognize and Name

When refactoring or reviewing, Codex SHOULD identify smells by their canonical names from Martin Fowler's *Refactoring* and Metz/Owen's *99 Bottles of OOP*. Naming the smell makes the diff reviewable and makes the cure obvious.

| Smell | Symptom | Typical cure |
|---|---|---|
| **Duplicated Code** | Same logic in 2+ places | Extract a function — but only after the duplication has a clear name |
| **Long Method** | Exceeds the 30-line hard limit | Extract helpers along responsibility lines |
| **Long Parameter List** | 4+ positional params, or unrelated params | Group into a dataclass; pass an object |
| **Switch Statements / `if/elif` on a discriminant** | Branches dispatch on type, kind, or status | Polymorphism, dispatch dict, or one class per case |
| **Primitive Obsession** | Domain concepts modeled as `int`, `str`, `dict`, `tuple` | Extract a class (often a frozen dataclass with methods) |
| **Data Clump** | The same 3+ values travel together as parameters or fields | Group into a single object |
| **Feature Envy** | A method uses another object's data more than its own | Move the method to that object |
| **Shotgun Surgery** | One logical change forces edits across many files | A missing abstraction; consolidate it |
| **Divergent Change** | One module changes for many unrelated reasons | Split it along the change axes |
| **Speculative Generality** | An abstraction with no second user | Delete; restore the concrete form |
| **Inappropriate Intimacy** | Two modules know each other's internals | Push behavior into the data owner |
| **Comments as Deodorant** | A comment explaining *what* the code does | Rename or extract until the comment is unnecessary |

When removing a smell, Codex SHOULD name it in the explanation/PR description, e.g.:

> Removes Primitive Obsession: extracts `BottleNumber` from `int` arguments shared by `container`, `quantity`, `pronoun`, and `successor`.

---

## OOP Guidance

Codex should not force OOP everywhere.

Use OOP when it improves:

- cohesion
- encapsulation
- reuse
- testability
- clarity of domain behavior

Good candidates for OOP:

- related operations over the same state
- domain concepts with behavior
- repeated workflows with shared dependencies
- services that coordinate external systems
- validation and transformation logic that belongs together

Codex SHOULD use:

- dataclasses for structured data
- frozen dataclasses for immutable config/field constants and value objects
- small cohesive classes with clear responsibilities

Codex MUST NOT:

- create classes only to group unrelated functions
- create large "manager" classes with mixed responsibilities
- hide procedural code inside a class without improving structure
- mix unrelated concerns in the same class

If OOP is chosen, class methods should follow the same 20-line target and 30-line hard limit.

### Immutability for value objects

Prefer constructing new instances over mutating existing ones for value-like types. Reach for mutability only at boundaries (caches, I/O state, request-scoped accumulators).

Frozen dataclasses make equality, hashing, testing, and reasoning straightforward.

## When to Extract a Class

Before extracting a class, run these diagnostics. **Three or more "yes" answers** strongly suggest the extraction is real, not speculative.

1. **Squint test**: blur your eyes — do several methods have the same shape? (same conditional structure, same argument, similar size)
2. Do those methods take an **argument with the same name**, representing the same concept?
3. Does that argument carry the bulk of the logic — i.e. the methods depend more on the **argument** than on `self`?
4. If you drew a private/public line through the class, would the same group of methods fall on one side?
5. Is the argument a **primitive** (`int`, `str`, `dict`) standing in for a domain concept (Primitive Obsession)?

If yes: extract a class named after **what the argument represents** — `BottleNumber`, `EmailAddress`, `MoneyAmount` — and move those methods onto it.

Steps for a safe Extract Class refactor:

1. Create the new (empty) class.
2. Copy the obsessing methods over without deleting the originals.
3. Put the new class into use from the call sites that have a tested safety net.
4. Once green, delete the now-unused originals.

Existing tests for the parent class become integration tests for the new class until dedicated tests are added.

## Refactoring Discipline

Refactoring changes the **arrangement** of code without changing **behavior**. Adding behavior and refactoring MUST be separate steps.

### Open/Closed first

Before adding new behavior to existing code, Codex SHOULD ask:

1. Is the code already **open** to this change? If yes, just write the new code.
2. If not, refactor to openness **first**. Then add the new behavior in a separate step (and ideally a separate commit).

Never mix "make room" and "add feature" in the same diff.

### Smell-driven refactoring

When the path to openness isn't obvious, follow code smells:

1. List the things wrong with the code.
2. Pick the most prominent smell.
3. Apply its standard cure (see the smells table above).
4. Re-evaluate. The path to openness usually reveals itself within a few smells removed.

### Flocking Rules — converging on an abstraction

When duplicated branches *almost* line up, use the Flocking Rules to discover the abstraction instead of guessing it:

1. **Select the things that are most alike.**
2. **Find the smallest difference between them.**
3. **Make the simplest change that removes that difference.**

Replace differing concrete values with a common variable or message send. Difference becomes sameness, and the new name describes the abstraction.

### Discipline of small steps

- Change **one line at a time** when refactoring.
- Run tests after each change.
- If a test fails, revert and make a smaller change.
- Never modify tests during a refactor. If tests assert *how* something works rather than *what* it does, fix the tests in a separate prior diff.

Tests are the safety net. Refactoring leans on green.

### When touching legacy or AI-generated code

Codex SHOULD improve nearby quality issues, especially:

- missing type hints
- vague names
- long functions
- repeated string literals
- duplicated logic
- excessive nesting
- unclear boundaries between data access and business logic

Codex MUST NOT leave obviously low-quality patterns untouched in code it is already editing, unless changing them would create an unsafe or oversized diff.

When a larger refactor is needed, Codex should propose a staged approach.

## Code Style Conventions

- Line length: 88 characters unless the project config says otherwise
- Quotes: double quotes
- Indentation: spaces
- Imports: sorted by ruff
- Prefer `pathlib` over `os.path`
- Prefer clear conditionals over clever comprehensions
- Prefer explicit return types
- Prefer small pure helpers where possible
- Prefer equality (`==`) over inequality/range checks when branching on a discriminant — it narrows the truth condition and makes future refactors safer

# Part 3 — Toolchain & Workflow

## Toolchain

This project uses **uv** as its sole package and environment manager.

Codex MUST NOT use:

- `pip`
- `pip-tools`
- `poetry`
- `conda`
- manually created virtual environments
- manually activated virtual environments

`uv` owns `.venv/` entirely.

---

## Package Operations

| Intent | Command |
|---|---|
| Add a runtime dependency | `uv add <package>` |
| Add a dev-only dependency | `uv add --dev <package>` |
| Remove a dependency | `uv remove <package>` |
| Sync the environment to the lockfile | `uv sync` |
| Regenerate the lockfile | `uv lock` |

Codex MUST modify dependencies only through `uv add` / `uv remove`.

Codex MUST NOT edit dependency lists in `pyproject.toml` by hand.

---

## Running Python and Tools

Always prefix project execution with `uv run`.

Direct invocations are forbidden because they may resolve to the wrong environment or system binary.

Forbidden:

```bash
python script.py
pytest
ruff check .
mypy .
```

Required:

```bash
uv run python script.py
uv run python -m module_name
uv run pytest
uv run ruff check .
uv run ruff format .
uv run mypy .
```

Use `uvx` only for one-off tools that are not project dependencies:

```bash
uvx <tool>
```

---

## Project Initialisation

For a general application or script:

```bash
uv init project-name
```

For a distributable library or package:

```bash
uv init --package project-name
```

Project metadata lives in `pyproject.toml` using PEP 621.

Codex MUST NOT create:

- `setup.py`
- `setup.cfg`
- `requirements.txt`

unless the user explicitly asks for compatibility files.

---

## Testing

Use pytest.

Rules:

- Entry point: `uv run pytest`
- Test root: `tests/` at the project root
- Test files: `test_*.py`
- Test functions: `test_*`
- `tests/__init__.py` is not required

Useful commands:

```bash
uv run pytest
uv run pytest tests/test_api.py
uv run pytest tests/test_api.py::test_specific_case
uv run pytest -v -x -s
uv run pytest --cov=mypackage --cov-report=term-missing
```

If pytest is missing and tests are needed:

```bash
uv add --group test pytest pytest-cov
```

Recommended pytest config:

```toml
[tool.pytest.ini_options]
testpaths = ["tests"]
addopts = "-v --tb=short"
```

Tests should assert **what** the code does, not **how** it does it. Implementation-coupled tests block refactoring; behavior-asserting tests enable it.

---

## Linting and Formatting

Use ruff for linting and formatting.

```bash
uv run ruff check .
uv run ruff check --fix .
uv run ruff format .
uv run ruff format --check .
```

Configuration should live under `[tool.ruff]` in `pyproject.toml`.

Recommended rule set:

```toml
[tool.ruff.lint]
extend-select = [
    "F",        # Pyflakes — undefined vars, unused imports
    "E", "W",   # PyCodeStyle — PEP 8
    "I",        # isort — import ordering
    "UP",       # PyUpgrade — modernise syntax
    "C4",       # Comprehensions
    "FA",       # Future annotations
    "ISC",      # Implicit string concat
    "RET",      # Return practices
    "SIM",      # Simplifications
    "PTH",      # pathlib over os.path
    "TC",       # Type-checking block
]
```

---

## Type Checking

Preferred type checker: **ty**, if configured.

Fallback: **mypy**.

```bash
uv run ty check
uv run mypy .
```

Codex MUST add type hints to:

- function arguments
- return values
- class attributes where useful
- dataclasses and typed structures

Never add a bare `# type: ignore`.

Allowed only with a specific error code and a reason where useful:

```python
value = external_call()  # type: ignore[attr-defined]
```

Preferred tool: **prek**.

# Part 4 — Safety & Reference

## Hard Constraints

The following actions are always wrong in this project:

**Toolchain:**

- Creating a virtual environment manually
- Activating a virtual environment manually
- Installing packages with `pip install`
- Using `requirements.txt` for dependency management
- Running `python setup.py`
- Adding dependencies directly to `pyproject.toml`
- Running project tools without `uv run`

**Code quality:**

- Adding a bare `# type: ignore`
- Creating large functions that exceed 30 lines
- Repeating structured string literals across the codebase
- Using vague function names for specific behavior
- Naming methods after their current implementation (e.g. `beer()` returning `"beer"`)
- Introducing abstractions, indirection, or "manager" classes without a concrete second user
- Mixing refactoring and behavior change in the same diff
- Modifying tests during a refactor

---

## Protect Against Compromised Packages

Codex MUST prefer a project-local `exclude-newer` setting in `pyproject.toml` when:

- creating a project
- scaffolding `pyproject.toml`
- updating an existing `[tool.uv]` table

Codex MUST add or merge:

```toml
[tool.uv]
exclude-newer = "14 days"
```

unless the user explicitly opts out or asks for a different interval.

Codex MUST preserve existing `[tool.uv]` settings when adding this guardrail.

Use `"14 days"` as the default cooling-off period.

Only use a machine-wide default if the user explicitly wants global uv configuration:

```toml
# ~/.config/uv/uv.toml
exclude-newer = "14 days"
```

Codex SHOULD briefly explain that this reduces the chance of installing newly published malicious packages before the community has time to detect and yank them.

## How To-s

### `uv run` vs `uvx`

| Use | Command | When |
|---|---|---|
| Project code, tests, apps | `uv run ...` | Needs project dependencies from `pyproject.toml` / lockfile |
| One-off or global-style tools | `uvx ...` | Tool should not depend on the project environment |

Examples:

```bash
uv run python script.py
uv run pytest tests/
uv run flask run

uvx ruff format .
uvx cookiecutter gh:user/template
uvx django-admin startproject mysite
```

### Python versions

| Goal | Command / Setting |
|---|---|
| Declare supported range | `requires-python = ">=3.11"` in `[project]` |
| Pin a single version | `uv python pin 3.12` |
| Override for one command | `uv run --python 3.10 <cmd>` |
| List available versions | `uv python list` |
| Install a specific version | `uv python install 3.12` |

`.python-version` controls the interpreter used for this project on the developer machine.

`requires-python` controls package compatibility for users.

The pinned version must satisfy `requires-python`.

### Virtual environment location

Default environment: `.venv/` at the project root.

| Method | Scope | How |
|---|---|---|
| `--active` flag | Single command | Activate a venv first, then `uv sync --active` / `uv add --active` |
| `UV_PROJECT_ENVIRONMENT=path` | Single command | `UV_PROJECT_ENVIRONMENT=custom-venv uv sync` |
| `.envrc` with direnv | Per-project | `export UV_PROJECT_ENVIRONMENT=custom-venv` |

Prefer the default `.venv/` unless the user asks for something else.

### Migrating from `requirements.txt`

```bash
uv init --bare
uv add -r requirements.txt
uv add --dev -r requirements-dev.txt
uv pip freeze
rm requirements.txt requirements-dev.txt
```

Do not use `requirements.txt` for ongoing dependency management.

### Optional dependencies vs dependency groups

| Feature | Optional dependencies | Dependency groups |
|---|---|---|
| Config section | `[project.optional-dependencies]` | `[dependency-groups]` |
| Published to PyPI | Yes | No |
| Install syntax | `pip install package[extra]` | `uv sync --group <name>` |
| Primary use | End-user feature sets | Dev tools |

Examples:

```bash
uv add --optional aws boto3 s3fs
uv add --group test pytest pytest-cov
uv add --group lint ruff mypy
uv sync --group test
uv sync --all-groups
```

### Troubleshooting

| Error | Cause | Fix |
|---|---|---|
| `No project table found` | `pyproject.toml` has no `[project]` section | Add `[project]` with `name` and `version`, or use `--no-project` |
| `.python-version` vs `requires-python` mismatch | Pinned version does not satisfy project requirement | `uv python pin <compatible-version>`, adjust `requires-python`, or use `uv run --python` temporarily |
| `ModuleNotFoundError` during install/build | Build isolation excludes dependencies not in `[build-system].requires` | Add missing dependencies to `[build-system].requires` |

## Final Self-Check Before Responding

Before finishing, Codex should check:

**Mindset & approach:**

- Did I preserve behavior unless explicitly asked to change it?
- Did I keep the diff reviewable and small?
- Did I separate refactoring from behavior changes?
- Did I avoid introducing speculative abstractions?

**Code quality:**

- Did I keep functions small (≤ 20 lines target, ≤ 30 lines hard limit)?
- Did I add type hints?
- Did I avoid vague names, and name methods after meaning rather than current behavior?
- Did I avoid repeated structured string literals?
- If I refactored, did I name the smell I removed?

**Toolchain:**

- Did I use `uv run` for project commands?
- Did I avoid direct `python`, `pytest`, `ruff`, or `mypy` calls?
- Did I run or suggest the relevant tests/checks?

If any answer is no, Codex should fix it or explicitly explain why.
